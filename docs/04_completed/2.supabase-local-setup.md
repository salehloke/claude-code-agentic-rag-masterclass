# Supabase Local Development Setup

✅ **Simple** — Well-documented CLI workflow, single-pass executable.

## Objective

Install the Supabase CLI and spin up the full local Supabase stack (Postgres, Auth, Storage, Studio, Realtime) via Docker, following the [official local development guide](https://supabase.com/docs/guides/local-development).

---

## Prerequisites

- [ ] Docker Desktop (or compatible runtime: OrbStack, Rancher Desktop, Podman) is installed and running
- [ ] Node.js 20+ installed (for npx usage) OR Homebrew available (for brew install)

---

## Tasks

### Task 1: Install Supabase CLI

Pick **one** method:

| Method | Command |
|:-------|:--------|
| Homebrew (macOS) | `brew install supabase/tap/supabase` |
| npm (dev dep) | `npm install supabase --save-dev` |
| pnpm | `pnpm add supabase --save-dev --allow-build=supabase` |

- [ ] Install the CLI using your preferred method
- [ ] Verify installation: `supabase --version`

**Validation**: `supabase --version` prints a version number without error.

---

### Task 2: Initialize Supabase in Project Root

- [ ] Run `supabase init` in the project root
- [ ] Confirm `supabase/` directory is created (contains `config.toml`)
- [ ] Ensure `supabase/` folder is safe to commit (it is — no secrets)

**Validation**: `ls supabase/config.toml` exists.

---

### Task 3: Start the Local Stack

- [ ] Run `supabase start` (first run downloads Docker images — may take a few minutes)
- [ ] Note the credentials output:
  - `API URL` (default: `http://localhost:54321`)
  - `DB URL` (default: `postgresql://postgres:postgres@localhost:54322/postgres`)
  - `Studio URL` (default: `http://localhost:54323`)
  - `Mailpit URL` (default: `http://localhost:54324`)
  - `anon key`
  - `service_role key`

**Validation**: Open `http://localhost:54323` in a browser — Supabase Studio loads.

---

### Task 4: Configure Environment Variables

- [ ] Create `.env` in project root (if not already present)
- [ ] Add the local credentials from `supabase start` output:
  ```
  SUPABASE_URL=http://localhost:54321
  SUPABASE_ANON_KEY=<anon key from output>
  SUPABASE_SERVICE_ROLE_KEY=<service_role key from output>
  SUPABASE_DB_URL=postgresql://postgres:postgres@localhost:54322/postgres
  ```
- [ ] Update `.env.example` with the same keys (no real values)
- [ ] Confirm `.env` is in `.gitignore`

**Validation**: `grep SUPABASE .env` shows all four keys populated. `.env` is not tracked by git.

---

### Task 5: Enable pgvector Extension

- [ ] Create a migration file: `supabase migration new enable_pgvector`
- [ ] Add to the generated migration SQL:
  ```sql
  create extension if not exists vector with schema extensions;
  ```
- [ ] Apply migration: `supabase db reset` (or restart with `supabase stop && supabase start`)
- [ ] Verify the extension is active:
  ```sql
  select * from pg_extension where extname = 'vector';
  ```

**Validation**: Query returns one row confirming `vector` extension is installed.

---

### Task 6: Verify Stack Health

- [ ] Run `supabase status` — all services show as running
- [ ] Connect to the database from a terminal:
  ```
  psql postgresql://postgres:postgres@localhost:54322/postgres -c "SELECT 1;"
  ```
- [ ] Verify Studio is accessible at `http://localhost:54323`

**Validation**: `supabase status` shows all services healthy, psql query returns `1`.

---

## Useful Commands Reference

| Command | Purpose |
|:--------|:--------|
| `supabase start` | Start local stack |
| `supabase stop` | Stop stack (preserves data) |
| `supabase stop --no-backup` | Stop stack and reset database |
| `supabase status` | Show running services and credentials |
| `supabase db reset` | Reset database and re-run all migrations |
| `supabase migration new <name>` | Create a new migration file |
